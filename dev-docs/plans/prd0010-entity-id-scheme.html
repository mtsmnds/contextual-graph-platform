PRD0010 — Entity ID Scheme & Model Rules

Final — validated against hello2/graph.json

This PRD merges the ID scheme design with the entity model rules discovered during the hello2/graph.json restructuring. The analysis log is at dev-docs/analysis/graph-model-decisions.md.

Problem

Current IDs are opaque integer prefixes: seg_18, act_11, note_1. With PRD0009, users own their data as plain JSON. An ID like seg_18 tells them nothing about what it refers to.

Part 1 — Entity Model Rules

Validated by restructuring 1359 entities in hello2/graph.json (Hamlet, Acts I–V, all scenes and speeches).

Rule 1: Segments never have a title field.

What was previously stored in title belongs elsewhere:

  • Character speeches → metadata.character (work-dependent field name)
  • Structural segments (stage directions, title page, etc.) → metadata.type drives rendering
  • Annotations → title is the card heading (annotations are the exception — they use title as a label)

Applied: 1132 character speech segments stripped of title and CHARACTER.\n content prefix. Zero data loss — every prefix was verified present before stripping.

Rule 2: Containers never have content.

Containers are structural labels only. Their title provides the heading; metadata provides rendering hints. Any inline content belongs in a child segment.

Applied: 20 scenes + 3 containers (title-page, about-the-ebook, dramatis-personae) had content moved to child segments. The child segment follows the standard counter scheme: seg-0001.

Rule 3: Entity deletion cascades to relations.

Removing an entity requires removing all relations referencing it and repairing the next chain. This is manual for now; future app-level delete will handle this automatically.

Rule 4: Entity rename propagates through all relations.

Renaming an entity requires updating every source/target in the relations array. Bulk find-and-replace. 38 relation references were updated for 6 renamed entities in the hamlet restructure.

Rule 5: metadata.type drives rendering.

The renderer determines display from metadata.type, not kind or title. kind (container vs segment) is a structural concern only. The metadata.type system is extensible — new types can be created freely, with the caveat that each new type adds renderer overhead. Evaluate with TipTap integration.

Rule 6: content supports HTML.

The content field can contain HTML for rich formatting. The renderer handles both plain text and rich content (detection: if content contains <, treat as HTML). This is important to add to the roadmap paired with TipTap implementation.

Part 2 — ID Scheme

Design Constraint: ID Stability

Critical tension: If IDs are derived from titles and containment, renaming a container changes every descendant ID. That breaks all source/target references in relations.

Rule: IDs are assigned once and never recalculated. The title/position slug is used at creation time to generate the ID. Renaming a node does not propagate to its children's IDs. Reordering segments does not renumber them. The ID is a snapshot of identity at creation time, not a live path.

This means:

  • A segment created as hamlet_act-i_scene-i_seg-0001 keeps that ID even if you rename the scene to "Scene 1 (Revised)"
  • A typo in a container title at creation time is permanent in the ID (edit the title field, the ID stays)
  • If you really need to change an ID, you edit graph.json directly and update all relations manually (future: app-level "rename entity" with automatic relation updates)

ID Format

Root container (no parent): [slug] Collision: append -1, -2, etc. Sub-container (act, scene, section, chapter): [parent-id]_[slug] If the parent uses positional counters (e.g. "Section 1", "Section 2"), use 2-digit padding: _sec-01, _sec-02 Segment (content-bearing, including container content children): [parent-id]_seg-[4-digit counter] 0001–9999 per parent container Annotation or Reference: [own-container-id]_ann-[slug] or [own-container-id]_ref-[slug] Not nested under the work — they live in their own containers like hamlet-notes, hamlet-references

Important clarification: Child segments that hold a container's content (scene descriptions, dramatis-personae list, title-page content, etc.) use the standard seg-[counter] format, not a _content suffix. The _content pattern was a temporary convention in the restructuring — the ID scheme always uses seg-0001.

Slug rules

  1. Lowercase the title
  2. Replace whitespace runs with -
  3. Strip punctuation (',.!?":;()[]{}—…,)
  4. Convert accented letters to ASCII (áéíóúçãõâêîôûñ → aeiouc aoaeioun)
  5. Collapse multiple - into one
  6. Trim leading/trailing -

Zero-padding pushback

Without padding: seg-1, seg-2, …, seg-10, seg-11.

The problem is sort order:

  • Alphabetical sort: seg-1, seg-10, seg-100, seg-11, seg-2, seg-20 — wrong
  • Natural sort: requires special handling in every tool (editors, git diff, scripting)
  • JSON arrays are ordered, but manual edits in a text editor reorder by alphabet

Zero-padded seg-0001 → seg-9999 sorts correctly alphabetically and numerically. The cost: slightly longer IDs and a hard cap at 9999 per parent (more than enough for any single container).

Recommendation: Keep padding. It solves a real (sorting) problem at negligible cost. 4 digits for segments, 2 digits for numbered sub-containers. Root containers don't pad (unique by name with collision suffixes).

Q&A

Q: What happens if I rename "Act I" to "Act One" — do child IDs change?

A: No. IDs are created once and never recalculated. Children keep their act-01 prefix. The title field is updated separately. IDs can become "stale" relative to titles — acceptable given the alternative (breaking all relations).

Q: What if I reorder segments — do counters renumber?

A: No. Reordering doesn't change IDs. Counters reflect position at creation/import time. The next relations encode order. The counter is an identifier, not a sequence number.

Q: Why not use UUIDs for stability?

A: Human-readability in the JSON file. A hash like a3f2c8e1 is stable but opaque. A semantic ID like hamlet_act-01_scene-01_seg-0001 tells you the entity's place without cross-referencing.

Q: Can the user change an entity's ID?

A: Not through the UI in v1. They can edit graph.json directly. Future: app-level "rename entity" that updates the ID and all relations referencing it.

Q: The _content child segments in hello2/graph.json don't follow the seg-0001 scheme — is that inconsistent?

A: Yes, that was a temporary convention. The hello2/graph.json file was restructured before the ID scheme was finalized. The child segments should eventually use seg-0001 in their IDs. The _content suffix works as a transitional identifier but is not the intended pattern.
