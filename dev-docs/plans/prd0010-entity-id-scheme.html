PRD0010 — Entity ID Scheme

Proposed — questions embedded

Problem

Current IDs are opaque integers with prefixes: seg_18, act_11, scene_14, note_1. They carry zero semantic information. With PRD0009, users now own their data as a plain JSON file. An ID like seg_18 tells them nothing about what it refers to. If they open graph.json in an editor, they have to cross-reference relations to understand the structure.

Semantic IDs fix this: the ID encodes the entity's place in the graph. hamlet_act-i_scene-i_seg-001 tells you immediately: this is the first segment in Scene I of Act I of Hamlet.

Goal

Define a deterministic, human-readable ID scheme that encodes the containment path. Replace all opaque IDs so the graph is self-documenting. Apply it to all future entities and migrate existing data.

Proposed Scheme

root: [slug] container: [parent-id]_[slug] segment: [parent-id]_seg-[counter] annotation: [annotated-id]_ann-[slug] concept: conc_[slug]

Slug rules

  1. Lowercase the title
  2. Replace whitespace runs with -
  3. Strip punctuation except apostrophes (they're content — who's)
  4. Collapse multiple - into one
  5. Trim leading/trailing -

Segment counter

Zero-padded index within the parent container. The number of digits should accommodate the largest container:

Parent typeMax segments (Hamlet)Suggested digits
Per scene~803 (001–999)
Per act~3003 (001–999)
Per work~13234 (0001–9999)

Hierarchy separator

_ descends one level. The ID is built by chaining parent IDs from root to leaf: 

work → work_act → work_act_scene → work_act_scene_seg-001

Questions

Q1: Root ID — long canonical title or short name?

The full work title slugifies to the-tragedy-of-hamlet-prince-of-denmark (39 chars). This propagates to every child ID:

Option A (full title)Option B (short canonical name)
the-tragedy-of-hamlet-prince-of-denmark_act-i_scene-i_seg-001 (71 chars)hamlet_act-i_scene-i_seg-001 (30 chars)

Option B uses a known short name (hamlet). For user-created projects the folder name could become the root slug automatically.

I'd recommend Option B — IDs are already long enough. The title is available in the entity's title field.

Q2: Segment counter scope — per-parent or sequential across the work?

Per-parent: counter resets within each container (scene → seg-001, seg-002…).

Sequential: single counter across the entire work (seg-0001, seg-0002…).

Per-parent gives more semantic signal (you can compute structure from IDs) but requires knowing the parent. Sequential is simpler to generate but encodes less information. The full path already contains the parent name, so the extra signal from per-parent counters is marginal.

I'd recommend per-parent with 3 digits (seg-001). No parent in Hamlet has more than 100 segments within a single scene.

Q3: Annotations with multiple parents — which parent in the ID?

An annotation like note_1 (Identity theme) annotates seg_18 (Barnardo's "Who's there?"). The annotation entity itself isn't contained by any scene — it's linked via annotates, not contains.

Options:

  • A: Use the annotated segment's ID as the prefix: …seg-001_ann-identity-theme
  • B: Annotations are root-level with an ann_ prefix: ann_identity-theme
  • C: Annotations use a slug derived from their own title with no parent: identity-theme

Option A ties the annotation to its source segment — but if an annotation links to multiple segments, which one goes in the ID? Option B/C are simpler and avoid the multiple-parent problem.

I'd recommend Option B — root-level with ann_ prefix. Clean and unambiguous.

Q4: Non-parented entities (dramatis personae, front matter) — what path?

In Hamlet, there are entities outside the act/scene hierarchy: the title page, Dramatis Personæ, Transcriber's Notes. These are contained by the work directly, not by an act. The ID would just omit the intermediate levels:

dramatis_5 (Dramatis Personæ)hamlet_dramatis-personae
seg_1 (title page)hamlet_seg-001 (first segment under work directly)

Multiple hierarchy depths in the same work: some entities go work→act→scene→segment, others go work→segment. This is fine — the ID reflects the actual containment path.

Q5: Stage directions with character metadata — no title, how to identify?

Many segments (speeches) have a character metadata field but no title. The content starts with the character name. Some segments (stage directions, scene descriptions) have neither title nor character.

Since segments use seg-[counter] regardless, this isn't an issue — the counter is the identifier. Character info stays in metadata.

Q6: Migration — rewrite hamlet.json or generate on import?

With PRD0009, hamlet.json is no longer bundled. But we still have it in the source tree as a reference. Two approaches:

  • A: Rewrite hamlet.json with new IDs, update all relations to match. The file becomes a sample data set using the new scheme.
  • B: Keep hamlet.json as-is. Only apply the new scheme to entities created through the app going forward. Build a migration script for users who want to update existing files.

I'd recommend Option A — the old IDs are a dead pattern. Rewriting hamlet.json is a one-time effort and gives us a validated reference implementation of the new scheme.
