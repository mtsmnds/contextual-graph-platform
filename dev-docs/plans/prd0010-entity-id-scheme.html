PRD0010 — Entity ID Scheme

Proposed — v2 after review

Problem

Current IDs are opaque integer prefixes: seg_18, act_11, note_1. With PRD0009, users own their data as plain JSON. An ID like seg_18 tells them nothing. Semantic IDs encode the entity's place in the graph — hamlet_act-i_scene-i_seg-0001 is self-documenting.

Design Constraint: ID Stability

Critical tension: If IDs are derived from titles and containment, renaming a container changes every descendant ID. That breaks all source/target references in relations.

Rule: IDs are assigned once and never recalculated. The title/position slug is used at creation time to generate the ID. Renaming a node does not propagate to its children's IDs. Reordering segments does not renumber them. The ID is a snapshot of identity at creation time, not a live path.

This means:

  • A segment created as hamlet_act-i_scene-i_seg-0001 keeps that ID even if you rename the scene to "Scene 1 (Revised)"
  • A typo in a container title at creation time is permanent in the ID (edit the title field, the ID stays)
  • If you really need to change an ID, you edit graph.json directly and update all relations manually (future: app-level "rename entity" with automatic relation updates)

Proposed Scheme

Root container (no parent): [slug] Collision: append -1, -2, etc. Sub-container (act, scene, section, chapter): [parent-id]_[slug] If the parent uses positional counters (e.g. "Section 1", "Section 2"), use 2-digit padding: _sec-01, _sec-02 Segment (content-bearing): [parent-id]_seg-[4-digit counter] 0001–9999 per parent container Annotation or Reference: [own-container-id]_ann-[slug] or [own-container-id]_ref-[slug] Not nested under the work — they live in their own containers like hamlet-notes, hamlet-references

Slug rules

  1. Lowercase the title
  2. Replace whitespace runs with -
  3. Strip punctuation (',.!?":;()[]{}—…,)
  4. Convert accented letters to ASCII (áéíóúçãõâêîôûñ → aeiouc aoaeioun)
  5. Collapse multiple - into one
  6. Trim leading/trailing -

Zero-padding pushback

Without padding: seg-1, seg-2, …, seg-10, seg-11.

The problem is sort order:

  • Alphabetical sort: seg-1, seg-10, seg-100, seg-11, seg-2, seg-20 — wrong
  • Natural sort: requires special handling in every tool (editors, git diff, scripting)
  • JSON arrays are ordered, but manual edits in a text editor reorder by alphabet

Zero-padded seg-0001 → seg-9999 sorts correctly alphabetically and numerically. The cost: slightly longer IDs and a hard cap at 9999 per parent (more than enough for any single container).

For sub-containers that are positional (like Chapter 1, Section 2), 2-digit padding (ch-01, sec-02) gives correct sort order up to 99. Named sub-containers (like dramatis-personae) don't need counters.

Recommendation: Keep padding. It solves a real (sorting) problem at negligible cost. 4 digits for segments because that's the largest volume tier. 2 digits for sub-containers because you rarely have more than 99 chapters/sections. Root containers don't pad because they're unique by name with collision suffixes.

Notes on specific examples

  • Root container: hamlet--william-shakespeare → hamlet. The full title "THE TRAGEDY OF HAMLET, PRINCE OF DENMARK" is the title field on the entity. The ID is a canonical short name. If another work called "Hamlet" appears, it becomes hamlet-1.
  • The title page ("THE TRAGEDY OF HAMLET, PRINCE OF DENMARK by William Shakespeare") is a segment within the hamlet container, not a separate container.
  • Dramatis Personæ is a named sub-container of hamlet: hamlet_dramatis-personae. No act/scene path.
  • Annotations/References live in their own containers (e.g., hamlet-notes), not under the work's act/scene hierarchy. Their IDs are hamlet-notes_ann-slug or hamlet-references_ref-slug.

Scope

This PRD covers: the ID scheme design, slug rules, counter scheme, and the ID generation utility function. It applies to all new entities created through the app.

Not in scope: Migrating hamlet.json to the new scheme (separate PRD). Hamlet keeps its old IDs for now. A future PRD will restructure the hamlet data with corrected containment and the new IDs.

Q&A

Q: What happens if I rename "Act I" to "Act One" — do child IDs change?

A: No. IDs are created once and never recalculated. The children keep their act-01 prefix. The title field on the container is updated separately. This means IDs can become "stale" relative to titles — but that's acceptable given the alternative (breaking all relations).

Q: What if I reorder segments — do counters renumber?

A: No. Reordering doesn't change IDs. Counters reflect position at creation/import time. If you need a specific order, you use next relations (which already encode order). The counter is an identifier, not a sequence number.

Q: What about entities created by agents (import scripts, automated tools)?

A: Same rules apply. The import script calls generateEntityId which uses the same slug + counter logic. Deterministic input → deterministic ID. Importing the same data twice produces the same IDs (minus collision suffixes for duplicates).

Q: Why not use UUIDs or hashes for stability?

A: The goal is human-readability in the JSON file. A hash like a3f2c8e1 is stable but opaque. A semantic ID like hamlet_act-01_scene-01_seg-0001 tells you the entity's place in the graph without cross-referencing. The trade-off is that renaming a container creates a mismatch between the ID and the title — but you can still trace the ID through the containment chain.

Q: Can the user change an entity's ID?

A: Not through the UI in v1. They can edit graph.json directly (it's their file). Future: an app-level "rename entity" action that updates the ID and all relations referencing it.
