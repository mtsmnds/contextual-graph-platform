# PRD0014 — TipTap UI Phase 1: Simple Editor Scaffold

**Date:** 2026-05-15
**Status:** Complete
**Milestone:** M2 — Reading Workspace (Current Sprint)

---

## Context

The first TipTap integration (PRD0013) wired a bare read-only editor into the reading viewport. Editing was possible via a minimal `RichTextContent` with `editable` prop, but there was no toolbar, no formatting controls, no image upload, and no link management. The user could type and use keyboard shortcuts, but the editing experience was indistinguishable from a bare contenteditable.

---

## Decision

Scaffold TipTap's Simple Editor template (MIT licensed) and use it as the editing surface for empty containers (Playground, Roadmap). The Simple Editor provides a full toolbar, all standard formatting, image upload, link popover, and dark/light mode support out of the box.

### In scope

1. Install SCSS support (`sass`) — required by TipTap UI Components
2. Scaffold Simple Editor via `npx @tiptap/cli@latest add simple-editor` — 144 files created
3. Create `TiptapEditor` wrapper component with content/onSave props
4. Wire into Playground — empty containers get the full editor, no duplicate content, no empty sidebar
5. Implement debounced save strategy (onBlur + 1.5s idle + onUnmount) to prevent ProseMirror/Zustand transaction collision

### Out of scope

- BubbleMenu / FloatingMenu (Phase 2)
- Drag Handle (Phase 2)
- Slash commands (Phase 2 — decision gate: free vs paid)
- Additional extensions: Underline, Highlight, TextAlign, TaskList, Placeholder, Emoji (Phase 2)
- Schema testing / ProseMirror JSON (Phase 3)

## ADR: Debounced Save Strategy

### Context

The initial `onUpdate` handler called `useGraphStore.getState().updateEntity()` on every TipTap transaction. This caused the Zustand store to update synchronously during ProseMirror's transaction cycle, triggering app-wide re-renders. Complex multi-step operations (like `toggleBlockquote` which wraps content, clears nodes, and repositions the cursor) could collide with React re-renders, causing the entire editor to crash.

### Decision

Replace `onUpdate`-driven immediate saves with a three-trigger save strategy:

1. **On blur** (`handleDOMEvents.blur`) — immediate save when the user clicks outside the editor
2. **On idle** (`onUpdate` with 1.5s debounce) — saves after the user stops typing for 1.5s
3. **On unmount** (`useEffect` cleanup) — saves when the user navigates away

The editor never calls into Zustand during active editing. The store is updated only when the editor is idle or losing focus.

### Alternatives Considered

- **Immediate save with queueMicrotask** — deferred the Zustand call but still ran within the same transaction cycle, potential for race conditions
- **Save on blur only** — simplest but loses content on browser crash
- **No Zustand sync at all** — requires a separate persistence layer for editor content, over-engineering for MVP

### Consequences

- Blockquote and other complex operations no longer crash the editor
- Typing feels smoother (no re-render on every keystroke)
- Content is at most 1.5s stale in the store (acceptable for single-player)
- On unexpected close (browser crash, power loss), at most 1.5s of work is lost
- Foundation for Phase 2 editing features (BubbleMenu, Drag Handle, extensions)
